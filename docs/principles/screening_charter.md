# Screening Charter

## Document Purpose

这份文件定义港股价值筛选系统的长期准则。

它不是行情评论，也不是一次性的选股想法，而是整个系统的工作宪法。

任何筛选规则、阈值、例外、经验教训、复盘结论，都必须和这份文件保持一致。

## Core Objective

第一轮筛选的目标不是“找出可以买的股票”，而是：

1. 快速排除明显不值得花时间研究的公司。
2. 把全市场缩小到值得进一步翻阅和理解的观察池。
3. 用一致的方法减少情绪化、主观化和熟悉度偏差。
4. 为后续“按字母翻阅”保留高质量候选对象。

## What Good Screening Looks Like

好的筛选，不是筛出最多股票，而是筛出最值得继续看的一批股票。

好的第一轮筛选应该具备：

1. 可重复：今天和下周跑，逻辑一致。
2. 可解释：每个阈值都能说清楚为什么存在。
3. 可修正：发现错误后能改，而不是硬撑。
4. 可追踪：任何改动都有日期、原因和结果。
5. 可分层：总规则稳定，行业例外单独处理。

## What Bad Screening Looks Like

以下都属于坏筛选：

1. 只因为低 `PE` 就保留。
2. 只因为低 `PB` 就认为安全。
3. 忽视现金流、负债、流动性和行业结构。
4. 规则今天一套、明天一套，没有版本记录。
5. 发现规则失效后不记录，只凭感觉修补。

## First-Pass Filtering Principles

第一轮筛选优先做“排除”，不是“证明优秀”。

### Default Exclusions

默认优先排除以下对象：

1. 股价过低的仙股和低价投机股。
2. 长期亏损或盈利极不稳定的公司。
3. 日均成交额过低、研究和交易都不友好的公司。
4. 明显依赖讲故事、资产质量不清楚的公司。
5. 业务模式无法在一段话里解释清楚的公司。

### Default Preferences

第一轮优先保留以下特征：

1. 估值不高。
2. 盈利为正。
3. `ROE` 或 `ROIC` 不差。
4. 市值和流动性达到基础要求。
5. 有持续分红或现金流支持。

## Baseline Screening Dimensions

每次第一轮筛选，至少检查这些维度：

1. 价格
2. 市值
3. 盈利状态
4. 估值
5. 资本回报率
6. 负债或杠杆水平
7. 流动性
8. 分红和现金流质量

## Blacklist Policy

黑名单是正式规则，不是临时备注。

只要一个对象被加入黑名单，后续扫描默认跳过，直到你主动移除或停用。

适合加入黑名单的对象包括：

1. 你明确不想继续扫描的公司
2. 重复证券类型，例如权证、预托、特殊类别证券
3. 长期制造噪音、但不在当前研究边界内的对象
4. 已经多次证明不符合你方法论的公司

黑名单必须记录：

1. 股票代码
2. 名称
3. 加入日期
4. 原因
5. 是否仍然生效

黑名单文件位置：

- `rules/blacklists/default.yaml`

## Sector Exceptions

以下行业不能机械套统一阈值，必须单独看：

1. 银行
2. 保险
3. 地产
4. 资源和能源
5. 强周期制造业

原因是这些行业的：

- 负债结构不同
- 利润周期不同
- `PB` 和 `PE` 解释方式不同
- 资本开支和分红模式不同

因此，主规则保留，行业附加规则另写。

估值规则不能在全市场用单一阈值硬套。

例如：

- `PB` 对银行更有意义
- `PB` 对互联网通常意义较弱
- 地产长期低 `PB` 可能只是行业常态
- 周期股低 `PE` 可能对应利润高点

行业分层筛选框架见：

- `docs/playbooks/sector_screening_framework.md`

## Alphabetical Review Principle

按字母或拼音首字母翻阅的作用，是扩大覆盖面，降低“只看熟悉公司”的偏差。

执行时应遵守：

1. 先过基础筛选，再按字母翻阅。
2. 翻阅阶段只找“值得深看”的线索，不直接下买入结论。
3. 看到异常便宜的公司，先问“为什么便宜”，再问“会不会错杀”。

## Rule Change Policy

任何筛选规则变更，都必须记录到变更日志。

每次修改至少写清楚：

1. 更新时间
2. 更新内容
3. 为什么改
4. 改好在哪里
5. 暴露了什么教训
6. 影响了哪些案例或股票
7. 是否需要继续观察

## Lessons Policy

教训不是附属品，是系统的一部分。

以下情况必须写教训记录：

1. 低估值但基本面很差的公司混进候选池。
2. 高质量但估值略偏高的公司被误杀。
3. 行业特性导致统一阈值失真。
4. 数据字段错误、缺失或解释错位。
5. 某条规则连续两次以上制造明显噪音。

## Iteration Cycle

建议每次规则迭代按这个顺序：

1. 跑筛选。
2. 按字母翻阅。
3. 抽样深看若干家公司。
4. 记录误判和漏判。
5. 更新规则或补充行业例外。
6. 在变更日志中追加记录。

## Current Baseline View

当前版本先坚持几个朴素原则：

1. 先远离明显垃圾，再谈便宜。
2. 先保证研究池可读，再追求覆盖全部机会。
3. 先用总规则，再逐步引入行业规则。
4. 规则宁可少，但必须稳定、可解释、可复盘。
